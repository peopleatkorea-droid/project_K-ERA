# Case Canvas Redesign Plan

## 목적

현재 `Case Studio / Case Canvas`는 기능은 많지만, 체감상 "문서를 작성하는 화면"보다 "관리 콘솔 안의 입력 폼"에 가깝다.  
이 문서의 목표는 다음 두 가지다.

1. 왜 현재 화면이 구식이고 덜 매력적으로 보이는지 구조적으로 설명한다.
2. 노션 계열의 현대적 문서형 UX를 참고하되, 의료 연구용 구조화 입력에 맞는 `structured document canvas` 설계안을 제안한다.

이 문서는 단순 비주얼 리프레시가 아니라, 케이스 작성 경험 전체를 다시 설계하는 기준안이다.

## 현재 진단

### 1. 방향은 문서형인데, 실제 체감은 대시보드형이다

프로젝트 문서상 의도는 이미 "document-style case canvas"다.  
하지만 실제 화면은 작성 캔버스보다 운영 대시보드의 문법을 더 많이 따른다.

- 좌측 레일에 병원, 진행도, 활동, 검증 정보가 크게 노출된다.
- 본문은 다시 큰 카드 묶음으로 나뉜다.
- 우측 패널도 별도 카드 스택으로 붙는다.

결과적으로 사용자는 "한 건의 케이스를 쓰는 중"이 아니라 "여러 위젯 사이에서 작업하는 중"처럼 느낀다.

### 2. 카드, 배지, 칩이 너무 많아 시선이 계속 끊긴다

현재 UI는 거의 모든 요소가 `rounded + border + surface` 조합이다.

- 섹션
- 서브 섹션
- 메트릭
- 상태 배지
- 리스트 항목
- 이미지 카드
- 토글 버튼

이 방식은 정돈되어 보이기는 쉽지만, 화면 전체가 비슷한 무게의 박스로 쪼개져 보여서 위계가 무너진다.  
요즘 앱처럼 보이지 않는 가장 큰 이유 중 하나가 바로 이 "박스 과다"다.

### 3. 입력 UX가 여전히 전통적인 폼이다

현재 `Field`는 정형 폼 컨트롤 중심이다.

- 라벨
- 보조 설명
- 테두리 있는 input/select/textarea
- 상태 메시지

이 방식은 관리 시스템에는 맞지만, "계속 써 내려가고 싶은 문서형 캔버스"의 감각과는 다르다.  
사용자는 콘텐츠를 작성하는 느낌보다 양식을 채우는 느낌을 받는다.

### 4. 문서형 캔버스인데 실제 흐름은 단계형 위저드에 가깝다

이미지 보드는 `intake_completed` 이후에만 열린다.  
즉, 캔버스처럼 보이지만 실제로는:

1. 기본 정보 입력
2. 완료 클릭
3. 그 다음 이미지 작업

순서가 강하게 고정되어 있다.  
이는 오류 방지에는 유리하지만, 사용자가 지금 어떤 재료를 갖고 들어왔는지에 따라 유연하게 시작할 수 없다.

### 5. 타이포와 컬러가 의도적으로 사용되지 않는다

폰트는 여러 개가 로드되어 있지만, 실제 캔버스에서는 거의 동일한 sans 계열 강도만 반복된다.  
제목, 본문, 메타 정보, 보조 설명의 톤 차가 크지 않아서 화면 전체가 "비슷한 온도"로 보인다.

컬러도 따뜻한 베이지 배경과 강한 블루 액센트를 사용하지만, 의료 문서 캔버스 기준으로는 다소 분리감이 있다.

- 배경은 부드럽다
- 액센트는 갑자기 강하고 디지털하다
- 카드와 칩은 다시 중성적이다

즉, 하나의 시각적 문법으로 수렴되지 않는다.

### 6. 저장 전 작성, 저장 후 리뷰, 환자 목록이 한 셸에서 경쟁한다

현재 워크스페이스는 아래 세 가지를 거의 한 덩어리 안에서 처리한다.

- 새 케이스 작성
- 저장된 케이스 리뷰
- 환자 목록 탐색

기능적으로는 강력하지만, 경험적으로는 한 화면의 정체성이 흐려진다.  
현대적인 앱은 보통 "현재 이 화면이 무엇을 위한 곳인지"가 매우 명확하다.

## 한 줄 결론

현재 화면이 구려 보이는 이유는 "스타일이 촌스러워서"보다,  
`문서 캔버스`를 표방하면서 실제로는 `폼 + 위젯 + 대시보드 + 위저드`가 한 화면에 겹쳐 있기 때문이다.

## 목표 방향

`Notion처럼 보이게`가 아니라, 아래 원칙을 만족하는 방향이 맞다.

1. 작성 중심: 현재 해야 할 일은 "한 건의 케이스 작성"으로 느껴져야 한다.
2. 문서 중심: 정보는 카드 모음보다 하나의 흐르는 문서처럼 읽혀야 한다.
3. 구조화 입력: 자유 텍스트 앱이 아니라 의료 입력이므로, 블록 안에서 구조화 입력을 자연스럽게 처리해야 한다.
4. 저마찰: 입력 전 긴 설명보다 바로 쓰고 붙이고 선택할 수 있어야 한다.
5. 점진 공개: 고급 옵션, 혼합균, 검증, 아티팩트는 필요할 때만 깊게 열린다.
6. 저장 안심감: 자동 저장, 로컬 초안, 상태 피드백이 조용하지만 분명해야 한다.

## 제안하는 제품 개념

### 새 이름/정체성

권장 개념명:

- `Case Canvas`
- 부제: `Structured Clinical Document`

핵심 메시지:

- 문서처럼 읽히고
- 폼처럼 검증되며
- 이미지 보드처럼 다룰 수 있어야 한다

즉, `Notion + structured medical intake + image workspace`의 조합으로 본다.

## 정보 구조 재설계

### 현재

- 좌측: 병원, 진행도, 활동
- 가운데: 리스트 또는 케이스 작성/리뷰
- 우측: 보조 패널

### 제안

작성 화면과 탐색 화면을 분리한다.

1. `Case List`
2. `Case Canvas`
3. `Case Review`

한 셸 안에서 토글하는 것은 가능하지만, 최소한 시각적으로는 전혀 다른 모드처럼 보여야 한다.

### 권장 레이아웃

#### A. Case List

- 목적: 환자/케이스 탐색
- 구성: 검색, 필터, 정렬, 환자 카드/행
- 톤: 데이터 브라우저

#### B. Case Canvas

- 목적: 새 케이스 작성 또는 기존 케이스 수정
- 구성: 단일 중심 컬럼 + 컨텍스트 우측 패널
- 톤: 문서 편집기

#### C. Case Review

- 목적: 저장된 케이스 검증, 기여, 아티팩트 리뷰
- 구성: 요약 헤더 + 타임라인 + 이미지 패널 + AI/validation rail
- 톤: 분석 워크벤치

## Case Canvas 상세 설계

### 1. 전체 레이아웃

권장 폭:

- 좌측 네비: 72px 아이콘 바 또는 접을 수 있는 220px 미니 사이드바
- 본문: 760px ~ 860px 단일 메인 컬럼
- 우측 컨텍스트: 300px ~ 340px sticky panel

핵심은 본문이 "문서 폭"으로 느껴져야 한다는 점이다.  
지금처럼 대형 카드가 넓은 공간을 채우면 콘솔처럼 보인다.

### 2. 상단 바

상단은 짧고 조용해야 한다.

- 좌측: 병원 이름 / 케이스 상태
- 중앙: 케이스 제목 또는 환자 식별자
- 우측: 자동 저장 상태, 최근 저장 시각, 더보기 메뉴

버튼은 최소화한다.

- `새 케이스`
- `목록으로`
- `제출`

이 정도만 상단 고정 액션으로 둔다.

`Momentum`, `Activity`, 병원 수준 메트릭은 작성 화면에서 제거한다.  
이 정보는 운영/리뷰 모드로 보내는 것이 맞다.

### 3. 본문 구조

권장 순서:

1. 케이스 헤더
2. 환자 스냅샷
3. 방문 블록
4. 원인균 블록
5. 임상 노트 블록
6. 이미지 보드 블록
7. 제출 전 체크 블록

문서처럼 이어지되, 각 블록은 접고 펼 수 있어야 한다.

### 4. 케이스 헤더

상단 첫 화면은 "양식"이 아니라 "문서 표지"처럼 보여야 한다.

포함 항목:

- 케이스 제목
- 병원명
- 초안 상태
- 작성자
- 마지막 저장 시각

예시:

`Case 24-031`
`Jeju Eye Center · Draft · Saved 8 sec ago`

제목은 크게, 메타는 작게.  
여기서부터 문서형 위계를 강하게 준다.

### 5. 환자 스냅샷 블록

지금의 `Patient identity` 섹션을 더 짧고 직접적으로 바꾼다.

구성:

- Patient ID
- Sex
- Age
- Local case code

중요한 점:

- 처음에는 한 줄 요약 상태로 보인다.
- 클릭하면 인라인 편집으로 확장된다.

예시 요약 상태:

`Patient 17635992 · Female · 65 · Case 24-031`

즉, 빈 폼이 아니라 "작성된 문장/레코드"처럼 보여야 한다.

### 6. 방문 블록

현재 `Visit context`와 `Visit timing`이 떨어져 있는데, 하나의 블록으로 합치는 편이 좋다.

블록 구조:

- Visit type: Initial / Follow-up
- FU number
- Visit date
- Visit status
- Contact lens
- Predisposing factors

여기서 중요한 것은 `칩 나열`보다 `선택 결과의 문장화`다.

예시:

- `Initial visit`
- `Contact lens: none`
- `Predisposing: trauma, steroid use`
- `Status: active`

편집 모드에 들어가면 segmented control, multi-select, date picker가 열리되, 기본 상태는 읽기 쉬운 문장 형태가 낫다.

### 7. 원인균 블록

이 블록은 현재도 구조는 맞지만, 표현이 너무 "설정 패널" 같다.

권장 구조:

- Primary organism
- Additional organisms
- Lab confidence or note

UX 원칙:

- 기본 선택은 `Primary organism` 하나에 집중
- `Mixed culture`는 보조 액션으로만 노출
- 추가 균주는 인라인 토큰으로 보이되, 편집시만 제거 핸들이 나타남

표현 예시:

`Primary: Fungal / Fusarium`
`Mixed: Curvularia, Candida`

### 8. 임상 노트 블록

현재 textarea는 너무 전통적이다.  
이 부분은 노션 감각을 가장 크게 만들 수 있는 영역이다.

권장 방식:

- 기본 높이는 1~2줄
- 포커스 시 자연스럽게 확장
- 플레이스홀더는 설명문이 아니라 문장 시작점

좋은 placeholder 예시:

- `Referral context, ocular surface status, procedural notes...`

나쁜 placeholder 예시:

- `Add concise clinical context for this visit.`

전자는 작성 유도를 하고, 후자는 입력 지시처럼 느껴진다.

### 9. 이미지 보드 블록

이 부분은 현재 "intake 완료 후 열림"인데, 작성 캔버스에서는 초반부터 보이는 것이 좋다.

권장 동작:

- 처음부터 이미지 블록이 보인다
- 단, 필수 메타가 부족하면 `Draft only` 상태로 표시
- 제출 전 검증에서만 필수 조건을 강하게 확인

즉, 입력 순서를 강제하기보다 제출 시점을 강제한다.

이미지 보드 구성:

- 상단: 대표 이미지 상태, 업로드 수, lesion box 진행도
- 본문: White lane / Fluorescein lane
- 하단: 대표 이미지 추천, 누락 상태, 저장 조건

시각적으로는 현재보다 훨씬 "갤러리/캔버스" 같아야 한다.

권장:

- 카드 수를 줄인다
- 이미지 자체를 더 크게 보여준다
- 이미지 위 액션은 hover 또는 selection 기반으로 노출한다

지금처럼 모든 카드 아래에 별도 버튼/배지/상태줄이 붙으면 조각난 느낌이 강하다.

### 10. 제출 전 체크 블록

마지막 블록은 요약 + 경고 + CTA다.

포함:

- 필수 입력 완료 여부
- 대표 이미지 지정 여부
- lesion box 작성 여부
- 저장 가능 여부

버튼 문구:

- `Save draft`
- `Submit to hospital`

가능하면 일반 저장은 autosave로 처리하고, 큰 CTA는 `Submit`만 남기는 편이 더 현대적이다.

## 우측 컨텍스트 패널 설계

우측 패널은 유지할 가치가 있다. 다만 역할을 바꿔야 한다.

### 현재 문제

- 요약, 메트릭, 히스토리, 기여, 검증이 한 톤으로 섞인다.

### 제안 역할

우측 패널은 "현재 문서의 상태"만 보여준다.

포함 권장:

- 저장 상태
- completeness
- 대표 이미지 상태
- AI suggestion
- validation readiness
- quick actions

작성 중에는 아래만 보여도 충분하다.

- `Draft saved`
- `4 / 6 sections completed`
- `Representative image missing`
- `1 image missing lesion box`
- `Run AI preview`

병원 단위 activity, validation history, contribution history는 `Case Review`에서 더 어울린다.

## 비주얼 시스템 제안

### 1. 톤

권장 무드:

- warm neutral canvas
- near-white document surface
- deep ink typography
- restrained clinical accent

즉, "병원 포털"보다 "고급 문서 워크스페이스"에 가까워야 한다.

### 2. 컬러

현재 블루 액센트는 조금 세다.  
권장 방향은 `stone + ink + deep teal + muted amber` 조합이다.

예시 토큰:

```css
:root {
  --canvas: #f6f3ee;
  --canvas-elevated: rgba(255, 252, 248, 0.88);
  --surface: #fffdf9;
  --surface-soft: #f2eee8;

  --text-strong: #16181d;
  --text: #2a2f37;
  --text-muted: #707784;

  --line: rgba(24, 28, 34, 0.08);
  --line-strong: rgba(24, 28, 34, 0.14);

  --accent: #0f766e;
  --accent-soft: rgba(15, 118, 110, 0.08);
  --warning: #b7791f;
  --danger: #c0565b;
}
```

핵심은 액센트가 화면 전체를 지배하지 않게 하는 것이다.  
강조는 적게, 상태는 조용하게.

### 3. 타이포

원칙:

- 제목은 크고 짧게
- 본문은 평평하고 읽기 쉽게
- 메타는 가볍고 촘촘하게

권장 역할 분리:

- UI font: `Noto Sans KR` 또는 현재 sans 체계
- Display/document heading: 현재 로드된 serif 계열을 케이스 제목에만 제한적으로 사용

지금처럼 많은 폰트를 로드하면서 대부분 같은 톤으로 쓰는 것보다,  
`주 폰트 1개 + 강조용 1개`가 훨씬 세련돼 보인다.

### 4. 테두리와 그림자

중요 원칙:

- 모든 요소에 border를 주지 않는다
- 가장 큰 면과 선택 상태에만 border를 쓴다
- 나머지는 배경 차와 여백으로 구분한다

권장:

- 문서 본문: border 거의 없음
- 블록 분리: 1px divider 또는 배경 톤 차
- 카드형 요소: 정말 필요한 경우만

### 5. 모션

의미 있는 모션만 남긴다.

- 블록 확장/축소
- autosave 상태 변화
- 이미지 선택/대표 지정
- hover reveal

금지:

- 모든 버튼에 과한 hover glow
- 패널마다 과한 부유감
- 불필요한 마이크로 애니메이션

## 입력 인터랙션 원칙

### 1. 기본은 읽기 모드, 클릭 시 편집 모드

각 블록은 처음부터 거대한 폼으로 보이면 안 된다.  
기본 상태는 요약 카드 또는 문장 블록으로 보이고, 클릭하면 그 안에서 편집 UI가 열리는 방식이 좋다.

### 2. Placeholder는 문장 시작점이어야 한다

`무엇을 입력해야 하는가`보다 `어떻게 쓰기 시작하면 되는가`를 제안해야 한다.

### 3. 라벨보다 내용 우선

필드 라벨은 상단 고정 텍스트 대신, 경우에 따라 inline prefix로 바꿔도 된다.

예시:

- `Contact lens: none`
- `Visit status: active`

이렇게 보이면 읽기와 편집이 자연스럽게 이어진다.

### 4. 선택지는 시각 밀도를 줄여야 한다

현재는 칩과 segmented control이 많다.  
좋은 방향은:

- 2지선다: segmented control 유지
- 다중 선택: inline token combobox
- 드문 고급 옵션: `+ Add mixed culture`처럼 접기

### 5. 저장은 조용하게, 제출은 분명하게

사용자는 계속 `저장` 버튼을 누르고 싶어 하지 않는다.  
좋은 경험은:

- 수정 즉시 autosave
- 상단에 `Saved just now`
- 문제 있으면 `Needs attention`

최종 CTA만 분명하게 둔다.

## 컴포넌트 레벨 설계

기존 `docs/css_design_system_plan.md`의 `token -> primitive -> section pattern` 방향과 맞춰 구현하는 것이 가장 안전하다.

### 신규/개편 primitive

1. `DocumentShell`
2. `DocumentHeader`
3. `CanvasBlock`
4. `InlineProperty`
5. `EditableSummary`
6. `ContextRail`
7. `StatusChip`
8. `ImageLane`

### CanvasBlock

역할:

- 제목
- 요약
- 본문
- 상태
- 접기/펼치기

예시 API:

```tsx
<CanvasBlock
  title="Visit"
  summary="Initial visit · active · no lens use"
  status="complete"
>
  ...
</CanvasBlock>
```

### InlineProperty

역할:

- 문장형 속성 표시
- 클릭 시 편집 가능

예시:

```tsx
<InlineProperty label="Contact lens" value="none" editable />
```

### ImageLane

역할:

- 뷰별 업로드 영역
- 이미지 썸네일 그리드
- 대표 지정
- lesion box 상태

핵심은 "입력 폼 하위 카드"가 아니라 "이미지 작업 레인"처럼 느껴지게 만드는 것이다.

## 화면별 리디자인 우선순위

### 1순위

- 작성 화면에서 병원 activity/momentum 제거
- 케이스 작성 본문을 단일 문서 컬럼으로 재배치
- `PatientVisitForm`를 블록형 요약/편집 구조로 변경
- `ImageManagerPanel`을 초기부터 보이게 변경

### 2순위

- 저장/리뷰 모드를 작성 모드와 시각적으로 분리
- 우측 패널을 작성 상태 중심으로 재정의
- 리스트 화면을 더 데이터 브라우저답게 정리

### 3순위

- AI preview와 validation 결과를 문맥형 패널로 고도화
- 키보드 흐름, 커맨드 메뉴, 빠른 추가 액션 도입

## 구현 단계 제안

### Phase 1. Quick win

1. 좌측 레일에서 작성과 무관한 메트릭 제거
2. 본문 최대 폭 축소
3. 카드 border 수 축소
4. 상단 헤더를 문서형으로 단순화
5. CTA를 `autosave + submit` 모델로 전환

효과:

- 코드 변경 대비 체감 개선이 가장 크다

### Phase 2. Canvas 구조화

1. `PatientVisitForm`를 `CanvasBlock` 기반으로 재구성
2. `Visit context`와 `Visit timing` 통합
3. 요약 모드/편집 모드 분리
4. 이미지 보드 선노출

효과:

- "폼 같다"는 인상이 크게 줄어든다

### Phase 3. Interaction polish

1. 인라인 편집
2. 조용한 autosave
3. 누락 상태 배지 개선
4. 이미지 hover 액션
5. 대표 이미지 추천/완료도 피드백

### Phase 4. Review 분리

1. 저장된 케이스 리뷰를 별도 시각 모드로 정리
2. 기여/검증/아티팩트 패널 재구성
3. 환자 타임라인을 분석형 레이아웃으로 정리

## 성공 기준

다음 질문에 "예"가 나오면 성공이다.

1. 처음 들어왔을 때 "문서를 쓰는 화면"처럼 보이는가
2. 어디부터 입력해야 할지 설명 없이도 감이 오는가
3. 빈 폼을 마주하는 부담이 줄었는가
4. 입력 결과가 문장과 요약으로 즉시 보이는가
5. 저장/제출 상태가 조용하지만 명확한가
6. 이미지 작업이 별도 하위 폼이 아니라 캔버스 일부처럼 느껴지는가

## 최종 제안

이 화면은 `예쁜 폼`으로 고치기보다,  
`관리 콘솔형 케이스 입력 화면`에서 `구조화 문서형 케이스 캔버스`로 제품 개념을 바꾸는 것이 맞다.

핵심 변경점은 세 가지다.

1. 작성 화면에서 운영 위젯을 빼고 집중도를 올린다.
2. 카드/배지/칩 중심 UI를 줄이고 문서형 블록 위계로 바꾼다.
3. 전통적 폼 입력을 요약 기반 인라인 편집으로 바꾼다.

이 세 가지만 해도 "요즘 앱처럼" 보이는 수준이 아니라,  
실제로 입력하고 싶고 오래 써도 피곤하지 않은 케이스 캔버스로 갈 수 있다.
